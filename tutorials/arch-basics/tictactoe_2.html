Game Architecture Basics

Rendering with OpenGL

The simplest way to render with JOGL is to register another callback interface: GLEventListener. Let's add the interface to our game class; register it with the window; and implement the interface's methods.

public class TicTacToeGame implements GLEventListener, WindowListener { ... window.addGLEventListener(this); ... @Override public void init(GLAutoDrawable drawable) { // called when a GL context is initialized } @Override public void reshape(GLAutoDrawable drawable, int x, int y, int width, int height) { // called when the window is moved or resized } @Override public void display(GLAutoDrawable drawable) { // called every frame to perform GL rendering } @Override public void dispose(GLAutoDrawable drawable) { // called when a context is being destroyed }

Each callback method receives a reference to the rendering surface in the form of a GLAutoDrawable. These methods are called on the thread that was created by the FPSAnimator, and the GL context is guaranteed to be current only inside these methods' stacks. That means if you try to make any GL call outside of these method stacks, it will fail. There's extra work you can do to make the context current outside of these methods, but I won't get into that right now. The simplest way forward is to make sure all our OpenGL work happens in these methods (or methods called by them.)

Clearing the Screen

Standard practice in OpenGL is to clear the screen at the start of every frame to make sure no extraneous data is left around to interfere with things.

public void display(GLAutoDrawable drawable) { // get the gl object out of the drawable GL4ES3 gl = drawable.getGL().getGL4ES3(); // set the clear color to red for visual feedback that it's happening gl.glClearColor(1, 0, 0, 0); // r, g, b, a // clear the color buffer to the stored clear color (red) gl.glClear(GL.GL_COLOR_BUFFER_BIT); }

And to make sure that the animator is constantly updating the screen we'll cycle the clear color between red and black.

GL4ES3 gl = drawable.getGL().getGL4ES3(); float i = (float) Math.sin(System.nanoTime() * 0.000000001) * 0.5f + 0.5f; gl.glClearColor(i, 0, 0, 0); gl.glClear(GL.GL_COLOR_BUFFER_BIT);
Rendering Vertices

Now we're ready to dive into rendering vertices. We'll create a Vertex Buffer Object to store vertex data, and we'll use a Vertex Array Object to inform OpenGL about the format and layout of that data.

But first...

What we've written so far has been straightforward procedural-style code to get the project started, but now we have our first real opportunity to talk about architecture. We want to build a rendering system that handles all of the rendering in the game and nothing else. That system will live in the game-architecture-basics module. It'll end up being the largest system in our game library.

When designing things we generally either start at the top (the big picture view) or the bottom (the small details). For a project like this I think it's important to understand how the details work before organizing them into larger components, so we'll start at the bottom for now. (Though it's common to work from both ends as well.) It makes a lot of sense because those bottom-level details are what you learn from beginner tutorials, and I'm writing this for the people who have done those and aren't sure what to do next.

So: vertex buffers and vertex arrays. It's easy to see that these are great constructs to wrap inside classes: they have data to be tracked and require operations to be performed on that data; they're easy to encapsulate and have very little dependence on the outside world; and we'll be reusing them a lot. These are the first tools we'll put in our toolbox.

First off is the VertexBuffer class. It goes in the game-architecture-basics module in our project. I've included the basic operations we'll want to do with a vertex buffer: binding and unbinding, putting new data into it, and deleting it. All these methods take the GL object as an argument to help signal that they should only occur on the render thread. I also indicate that in the Javadocs for each method. This is the last time I'll show those comments in the tutorial text, but you can assume I'll put that on every method that makes calls to OpenGL!

public class VertexBuffer { private int glId; private int target; private int usageHint; /** * Only call on the render thread! */ public VertexBuffer(GL4ES3 gl, Buffer data, int target, int usageHint) { this.target = target; this.usageHint = usageHint; int[] r = new int[1]; gl.glGenBuffers(1, r, 0); glId = r[0]; bind(gl); gl.glBufferData(target, data.capacity(), data, usageHint); unbind(gl); } /** * Only call on the render thread! */ public void update(GL4ES3 gl, int offset, int size, Buffer data) { bind(gl); gl.glBufferSubData(target, offset, size, data); unbind(gl); } /** * Only call on the render thread! */ public void bind(GL4ES3 gl) { gl.glBindBuffer(target, glId); } /** * Only call on the render thread! */ public void unbind(GL4ES3 gl) { gl.glBindBuffer(target, 0); } /** * Only call on the render thread! */ public void delete(GL4ES3 gl) { if (glId > 0) { gl.glDeleteBuffers(1, new int[]{glId}, 0); glId = 0; } } }

And the VertexArray class.

public class VertexArray { private int glId; public VertexArray(GL4ES3 gl) { int[] r = new int[1]; gl.glGenVertexArrays(1, r, 0); glId = r[0]; } public void bind(GL4ES3 gl) { gl.glBindVertexArray(glId); } public void unbind(GL4ES3 gl) { gl.glBindVertexArray(0); } public void delete(GL4ES3 gl) { gl.glDeleteVertexArrays(1, new int[]{glId}, 0); } }

Now we need to create some data to put in the vertex buffer and configure the vertex attributes. Then we can render! Let's go back into the game class and add a couple fields:

private VertexBuffer vertexBuffer; private VertexArray vertexArray;

Then we'll set them up in init():

GL4ES3 gl = drawable.getGL().getGL4ES3(); // a square from (-0.75, -0.66) to (0.5, 0.33) float[] verts = new float[] { -0.75f, -0.66f, -0.75f, 0.33f, 0.5f, -0.66f, 0.5f, 0.33f }; // Buffers is a JOGL helper class that makes sure buffers are configured correctly for your computer's architecture ByteBuffer buf = Buffers.newDirectByteBuffer(verts.length * 4); buf.asFloatBuffer().put(verts); vertexBuffer = new VertexBuffer(gl, buf, GL.GL_ARRAY_BUFFER, GL.GL_STATIC_DRAW); // bind the vertex array and vertex buffer before configuring the attribute, then unbind vertexArray = new VertexArray(gl); vertexArray.bind(gl); vertexBuffer.bind(gl); gl.glEnableVertexAttribArray(0); gl.glVertexAttribPointer(0, 2, GL.GL_FLOAT, false, 8, 0); vertexArray.unbind(gl); vertexBuffer.unbind(gl);

Now we just add a draw call after clearing the screen in display():

vertexArray.bind(gl); vertexBuffer.bind(gl); gl.glDrawArrays(GL.GL_TRIANGLE_STRIP, 0, 4);

When I run the program now I see a black square with the rest of the window still cycling between red and black. Success! Technically, OpenGL says that rendering with no bound shader program is not an error but has undefined results, so don't worry if you see something different. we'll tackle shader programs in the next section.
