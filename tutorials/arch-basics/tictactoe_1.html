Game Architecture Basics

Chapter 1 — Ultimate Tic Tac Toe

This first chapter gets us off the ground with a very simple game, Ultimate Tic Tac Toe. I'll start by explaining some of the most basic—and most important—design principles we'll use throughout the rest of the series. The game won't look like much, but that's intentional. I wanted to make something that would get us started as quickly as possible.

Design Principles

First let's define some terms. A Design Pattern is a well-defined, general solution to a common problem. Using established design patterns correctly is crucial when working on a development team. But even if you're going solo it's very important to understand and use these patterns. Doing so will make your codebase better (read: easier to work with), which will make your game development easier and more enjoyable.

But before we talk about full design patterns, we need to establish some principles. These are the guiding ideals for good code. Following these design principles keeps us focused on the qualities of "good code": organization, simplicity, correctness, robustness, efficiency, and so on. Design patterns are built according to these principles, so we have to understand the principles in order to utilize the patterns effectively.

Abstraction

If you take away nothing else from this series, learn this. Abstraction is the cornerstone of everything in computing. It's all about hiding complex details behind a simplified facade. Abstraction allows us to take a problem and break it down into simpler pieces. If we've abstracted a problem effectively, we'll have created a generalized tool that can be applied to other problems.

Low levels of abstraction are "closer" to your machine's hardware. The lower levels of your code know how to render simple shapes to your screen, or how to play sounds and manipulate audio.

High levels of abstraction help us see the big picture, because those low-level details are abstracted away, or hidden from view. Higher abstraction helps us keep a larger mental model of the whole system because we don't have to think about those low-level details if we've implemented the low-level systems correctly.

Java is a high-level language that follows an object-oriented Paradigm. That means its primary form of abstraction is the object. A Java Object is a bundle of code and data that serves a particular purpose. An object provides a public interface, which is how other objects interact with it. In other words, "public interface" is a term we use to refer to all of an object's publicly-accessible members.

Java defines objects through Classes and Interfaces. A class provides the definition of an object's type. A Car class in Java might declare that Car objects can have things like a make, a model, an engine type, a number of doors, or a color. You could then create objects (or instances) of that class for each car your program cares about. Those objects would then know which make, model, engine, color, etc that particular car actually has. So a class defines a type of thing, while an object represents a particular instance of that type.

Java interfaces define a set of publicly-accessible methods. An interface on its own does nothing. Its purpose is to be implemented by one or more classes. Classes that implement an interface are required to provide method definitions that match those declared by the interface. So the interface tells you what you can do with an object without needing to know exactly how that gets done. Multiple objects that correctly implement the same interface should work interchangeably.

If, for example, you're writing a game for multiple platforms, you could use interfaaces to hide the platform-specific aspects of your rendering system. The rest of the game's code only interacts with the rendering system through its interface, so if you simply provide a different implementation on different platforms you've successfully abstracted away the multi-platform details of rendering. You don't have to think about it any more.

Think of a car. A car is a complicated set of interacting systems that accomplishes a given purpose (locomotion.) It has an engine and fuel system, a transmission system, an electrical system, an exhaust system, a steering system, a braking system, a suspension system, and more. But to drive a car you don't need to know how most of those systems work. You need to understand how to use the accelerator, the brake, and the steering wheel. Those comprise the interface you use to interact with the whole system. Once you know how to use that interface you should be able to use any other machine that provides the same interface.

You may have noticed that example is incomplete, because there are actually two different interfaces for cars. I described the interface for cars with automatic transmissions. A manual transmission has two more methods you have to understand: the stick shift and the clutch. And because those methods can't be ignored, it has to be a separate interface. The two are not interchangeable.

Single Responsibility

This goes hand-in-hand with abstraction. The idea is that every distinct thing you create should serve a single purpose. It should have one single responsibility. It's the scope of those responsibilities that will change. You can think of your game as a stack of abstractions. At the top of the stack, the highest level of abstraction, is our Game class. It's responsible for running the entire game, but nothing else. It does that by using a Rendering system, an Input system, an Audio system, etc. It knows how to use these systems (through their interfaces), but it doesn't care how they work internally. Each of these systems in turn does only one thing within the Game. To do that they're composed of smaller parts that each do only one thing within that system. Get the idea? This goes all the way down to the lowest levels of our code. When you create a new system, or class, or function, make sure it does only one thing at the proper level of abstraction.

Code Reuse

This one is also known as Don't Repeat Yourself. If you find yourself writing a chunk of code you've written before in a similar context, it's time to refactor. Tuck away that duplicate code in a function and call the function in its place. If that isn't easy to do, you're probably looking at a good place to add a new abstraction or it's a symptom of poor design choices. But remember that note above about a similar context. You'll end up writing many, many for loops, for example. That doesn't mean you need to figure out how create a single abstraction so that you never have to write another for loop again. Otherwise we would already have that abstraction. But if you're writing multiple for loops to iterate over the same structure, that can almost certainly be hidden in an abstraction.

The Tutorial

With all those words out of the way, it's time to start writing code. Make sure you've installed Android Studio and know how to create a project and launch an app on your device. The developer docs are your friend.
